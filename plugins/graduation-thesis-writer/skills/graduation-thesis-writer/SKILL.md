# 毕业设计论文编写

## 使用原则

先读取 `references/format-rules.md`，并把其中的格式规范作为硬性约束。不要依赖用户原始参考格式文件；本技能已经内置目录、摘要、正文和参考文献规范。

最终产物应体现毕业设计论文的常见最佳实践：结构清晰、术语一致、论证闭环、代码与设计对应、图表服务于说明、引用可追溯、格式统一。不要为了凑字数堆叠代码清单或空泛描述。

## 标准工作流

1. 分析代码库并生成大纲 Markdown。
   - 先阅读用户指定的代码库入口、目录层级、配置文件、核心模块、数据流、接口、运行方式和依赖。
   - 按“项目背景-需求分析-总体设计-详细设计与实现-测试-总结”的毕业设计逻辑组织内容。
   - 生成 `毕业设计大纲.md`，标题层级最多规划到三级，后续 Word 正文以该大纲为唯一结构来源。
   - 大纲中每个小节写清“本节要回答的问题”和“需要从代码库取证的材料”，避免只列标题。

2. 按格式规范搭建 Word 正文骨架。
   - 根据大纲创建论文正文骨架，只放标题层级、章节占位说明、图表占位和参考文献占位，不填充完整正文。
   - 样式必须遵守内置规范：章标题、二级标题、三级标题、正文、引用、摘要、目录和参考文献分别设置。
   - 如果生成 `.docx`，优先使用结构化 Word API；需要运行 Python 时按当前项目要求使用 `uv run`。

3. 阅读代码库并按一级标题分批填充正文。
   - 以一级标题为最小交付批次，例如先完成 `1 绪论` 下的所有二级、三级标题，再进入 `2 系统相关理论与技术`。
   - 每批开始前只收集本章所需的代码证据、运行证据和引用资料，避免一次性展开过多章节导致质量下降。
   - 逐一对照正文骨架的小节写作，每节先确认代码证据，再形成论文叙述。
   - 需求分析写用户角色、功能需求、非功能需求和约束；总体设计写架构、模块划分、关键流程和数据结构；详细设计写核心模块实现逻辑；测试写测试环境、测试用例、结果和缺陷处理。
   - 对代码实现只摘取关键逻辑，不大段粘贴源码。必要时用流程图、模块表、接口表、数据库表或配置表解释。
   - 所有外部资料、标准、框架文档、论文或网络资料都必须进入参考文献，并在正文用方括号数字上标引用。

4. 每完成一个一级标题后暂停检查。
   - 完成当前一级标题下所有内容后，向用户交付本章内容、主要代码证据、已使用参考文献和需要确认的问题。
   - 明确等待用户检查确认；用户确认后再继续下一章。
   - 如果用户提出修改意见，先修订当前章，再继续后续章节。
   - 除非用户明确要求连续生成，否则不要一次性写完整篇正文。

5. 完成摘要、目录和终稿检查。
   - 正文基本稳定后再写中文摘要、英文 Abstract、关键词和 Key words。
   - 最后生成目录，目录最多列至三级标题，页码与正文分页一致。
   - 终稿检查标题层级、字体字号、首行缩进、引用编号、参考文献顺序、图表编号、页码和空白页。

## 分批写作交付规范

- 每个批次只覆盖一个一级标题及其全部下级标题。
- 每个批次完成后输出或更新 Word 中对应章节，并可同步维护章节 Markdown 草稿。
- 批次交付时列出本章已完成的小节、引用编号、图表占位或已生成图表、仍需用户确认的事实。
- 批次之间保持术语、章节编号、图表编号、参考文献编号连续一致。
- 后续章节不得改写已确认章节的核心结论；如必须调整，应先说明影响范围并征得用户确认。

## 写作质量要求

- 摘要应概括研究背景、目标、主要工作、实现方法、测试结果和结论，不写成项目介绍广告。
- 绪论应说明选题背景、意义、国内外相关现状、主要研究内容和论文组织。
- 需求分析应能直接支撑系统设计，避免和实现章节重复。
- 总体设计应先给架构与模块，再展开关键流程，不从零散函数开始写。
- 详细设计与实现应围绕核心业务链路、关键算法、关键接口、异常处理和安全性展开。
- 测试章节应包含可复现的环境、用例、输入、预期结果、实际结果和结论。
- 结论应总结完成情况、创新点或工程价值、不足和后续改进。
- 致谢、参考文献、附录等部分按学校要求保留；没有可靠来源时不要伪造文献。

## 文件与验证

推荐输出：

- `毕业设计大纲.md`：代码库分析后的层级化大纲。
- `毕业设计论文.docx`：按规范生成并填充的论文终稿。
- `论文生成说明.md`：可选，记录代码库证据、参考资料来源和未完成事项。

生成 Word 后应做一次真实渲染或打开检查，确认目录、摘要、正文标题、参考文献、分页和中文字体没有明显错乱。如果无法渲染检查，要明确告知风险。
